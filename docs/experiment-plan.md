# Experiment plan: Q-A interaction at PANL

## 1. Research decision

The paper should test one falsifiable question:

> Does PANL contain a question-answer interaction component that is separable from
> conditional answer likelihood and commitment, and is that component causally used for
> verbal confidence?

The target is not to show merely that PANL depends on both the question and answer. A
Transformer hidden state will normally be context-dependent. The evidence must jointly show:

1. **Identification:** a non-additive `Q x A` contrast on a pre-softmax confidence scale.
2. **Selectivity:** the component is not reducible to likelihood, identity, or commitment.
3. **Generalization:** it transfers to held-out questions, answers, and relation families.
4. **Causal use:** selective intervention changes the matched-vs-crossed confidence gap.

Three competing accounts remain live throughout the analysis:

- `H_fluency`: confidence is a function of conditional generation statistics `p(A | Q)`.
- `H_commit`: confidence primarily reads out readiness to commit rather than correctness.
- `H_relation`: confidence additionally uses a relational `psi(Q, A)` component.

Negative results are publishable if they establish that the apparent relational signal is
explained by fluency or commitment. Claims must be selected after the preregistered gates,
not after inspecting favorable plots.

## 2. Scope and model order

Run models in this order:

| Role | Model | Purpose |
|---|---|---|
| Smoke | `Qwen/Qwen2.5-0.5B-Instruct` | Validate token positions, hooks, schemas, and tests |
| Primary | `Qwen/Qwen2.5-7B-Instruct` | All experiments and fast iteration |
| Replication | `google/gemma-3-27b-it` | Confirm E1-E3 in a second model family |

Use greedy decoding and record the exact model revision, tokenizer revision, dtype, prompt,
chat template, seed, package lock hash, and git commit in every run manifest. Gemma is gated;
the remote worker needs `HF_TOKEN`, but secrets must never be written to configs or logs.

Local hardware has no detectable NVIDIA runtime and about 16 GB RAM. It is suitable for data
construction, tests, and analysis only. Plan GPU inference on Linux:

- Qwen 7B: one >=24 GB GPU is a practical starting point for BF16 inference and hooks.
- Gemma 27B: prefer >=80 GB VRAM or model-parallel execution.
- Quantization may be used for smoke tests, not for the primary causal result unless an
  unquantized replication shows the same direction and effect.

## 3. Data design

### 3.1 Crossed blocks

Each block contains two questions and their matched answers:

```text
(Q1, A1) matched       (Q1, A2) crossed
(Q2, A1) crossed       (Q2, A2) matched
```

Question and answer identities therefore occur once in each condition. Pair questions only
within a relation family and answer type. Do not split cells from one block across train and
test sets.

Required table: `data/processed/quadruples.parquet`.

| Column | Meaning |
|---|---|
| `block_id` | Stable identifier for the four cells |
| `relation_family` | Family held out for out-of-domain tests |
| `question_id`, `answer_id` | Identity controls |
| `question`, `answer` | Rendered text |
| `matched`, `correct` | Experimental labels; keep distinct where needed |
| `answer_source` | `gold`, `model_error`, `alias`, or `distractor` |
| `policy_status` | `natural`, `high_lp_wrong`, `low_lp_correct`, or `off_policy` |
| `template_id` | Prompt/template control |
| `split` | `train`, `validation`, `test`, or `family_holdout` |

### 3.2 Dataset tiers

1. **Tier 1, controlled facts:** capitals, currencies, authors, and inventors. Construct clean
   blocks from structured facts and render multiple question paraphrases.
2. **Tier 2, hard confusions:** near-neighbor entities, plausible wrong answers, rare correct
   aliases, and high lexical-overlap distractors.
3. **Tier 3, generalization:** at least one non-atomic family, initially passage-grounded
   answer verification or textual entailment. Two-hop QA is a stretch target.

Create two explicit stress subsets after scoring all candidates:

- high-likelihood wrong answers;
- low-likelihood correct answers.

Also build a naturalistic subset from errors actually generated by the model. Report the
teacher-forced controlled set and natural errors separately; do not claim that an off-policy
trajectory proves the same computation occurs during natural generation.

### 3.3 Split discipline

Use three increasingly strict evaluations:

1. unseen blocks with known relation families;
2. unseen question and answer identities;
3. unseen relation families.

Fit residualizers, PCA/SVD bases, probes, thresholds, and intervention scales on training data
only. Use validation for rank and scale selection. Open the test set once for final figures.

## 4. Measurements shared by all experiments

### 4.1 Positions

Resolve semantic positions from tokenized prompts, never from hard-coded token indices:

- `AC`: answer colon, before answer generation;
- `LAT`: last answer token;
- `PANL`: first newline token after the answer;
- `PANL+1`: immediate control position;
- `CC`: final confidence colon before confidence generation.

Every prompt must pass a tokenizer-level position test for both model families. Save token IDs
and resolved indices in each result row.

### 4.2 Behavioral outcome

The primary outcome is a pre-softmax confidence contrast. For categorical confidence, define a
fixed high-vs-low class logit margin. Keep class-token mappings fixed before experiments. For a
numeric prompt, use the logits for allowed first tokens and an ordinal latent-score model.

Reported numeric confidence is secondary because output nonlinearities can create a non-zero
difference-in-differences even when the latent computation is additive.

For block `k`, compute:

```text
I_k = z_11 - z_12 - z_21 + z_22
```

Report mean interaction, a standardized paired effect, sign consistency, block bootstrap
confidence intervals, and a within-block label permutation test. The block—not the cell—is the
independent resampling unit.

### 4.3 Conditional-likelihood controls

Collect teacher-forced statistics for each answer:

- token log-probability mean, min, max, and variance;
- first- and last-token log-probability;
- cumulative sequence NLL and length-normalized NLL;
- answer-token rank and entropy;
- answer-token margin over the top alternative;
- character length, token length, and tokenization pattern.

Residualize confidence and representations against these variables using cross-fitting. A
single in-sample regression coefficient is not sufficient evidence.

### 4.4 Commitment controls

Using the same completed answer state, elicit a commit/abstain decision in a separate prompt
condition. Train matched probes for relational fit, correctness, commitment, and likelihood.
Measure direction cosine similarity and re-run decoding/intervention after projecting out the
commitment and likelihood nuisance subspaces.

Ownership framing is a robustness analysis only; it must not be mixed into the main blocks.

## 5. Experiment matrix

### E0 - Pipeline reproduction and calibration

**Purpose:** validate that the implementation recovers the known PANL confidence-cache pattern
before testing a new mechanism.

- Reproduce confidence decoding and a small PANL steering/patching effect on the primary model.
- Confirm that effects emerge at PANL before CC and are weak at PANL+1/AC controls.
- Verify calibration, confidence-class tokenization, and exact prompt positions.
- Run sizes: 32 blocks for CI/smoke, then 200 blocks for the pilot.

**Gate E0:** do not interpret E1-E4 if no reliable confidence signal or PANL-localized effect is
present. Debug prompt fidelity and position resolution first.

### E1 - Factorial behavioral identification

**Purpose:** test whether confidence logits require a `Q x A` term after measured fluency.

- Compare additive and interaction models on pre-softmax logits.
- Include block/question/answer structure and template effects.
- Cross-fit likelihood residualization.
- Report controlled, natural-error, high-LP-wrong, and low-LP-correct subsets separately.
- Use the 200-block pilot for variance estimation; choose the main sample size with clustered
  simulation/power analysis rather than an arbitrary cell count.

**Gate E1:** advance the relational claim only if held-out interaction remains directionally
consistent after likelihood controls. Otherwise frame the paper as a fluency explanation.

### E2 - Layerwise factorial representation decomposition

**Purpose:** locate a selective, generalizing interaction component.

At each layer and position, compute the factorial residual:

```text
R_l = H_11^l - H_12^l - H_21^l + H_22^l
```

- Compare question main-effect energy, answer main-effect energy, and interaction energy.
- Learn a low-rank relational subspace from training residuals only.
- Evaluate matched-vs-crossed decoding on all three split levels.
- Quantify overlap with question identity, answer identity, likelihood, correctness, and
  commitment; then repeat after nuisance projection.
- Compare PANL with AC, LAT, PANL+1, and CC across all layers.

Store metadata/behavior as Parquet and activation shards as Zarr. Never accumulate all layers
on GPU or serialize a single monolithic tensor. Activation files must record model revision,
prompt hash, block IDs, dtype, shape, and checksums.

**Gate E2:** the candidate subspace must generalize to held-out identities and at least one
held-out relation family after nuisance removal. Mere in-sample probe accuracy is insufficient.

### E3 - Selective interaction-subspace intervention

**Purpose:** establish causal contribution while preserving main effects.

- Construct the relational projection on training data and freeze it.
- On held-out blocks, inject or sign-flip only the projected component at PANL.
- Test both directions and a preregistered scale grid chosen on validation data.
- Primary outcome: change in `Delta_fit = z_matched - z_crossed`, not a uniform shift in
  absolute confidence.
- Monitor commitment logits and nuisance decoders to show the effect is not wholly mediated by
  commitment or identity changes.

Required controls:

- matched-norm random subspace;
- orthogonal complement direction;
- shuffled block labels during subspace construction;
- PANL+1, AC, LAT, and CC position controls;
- neighboring layer controls;
- full-state swap as a sanity check only.

**Gate E3:** reserve the phrase "causal evidence" for a bidirectional, held-out effect that
exceeds random/orthogonal controls and survives likelihood/commitment residualization.

### E4 - Dual-input/path necessity

**Purpose:** test whether both question and answer information must reach PANL.

- Block or path-patch `Q -> PANL`, `A -> PANL`, and both paths.
- Compare with token-count- and layer-matched random attention-edge controls.
- Test whether `Delta_fit` contracts; a global confidence decrease alone does not support the
  relational account.

E4 is a stretch experiment for the main paper. Implement it only after E1-E3 pass their gates.

## 6. Planned repository architecture

```text
configs/
  data/                 # tier and split specifications
  experiment/           # E0-E4 run configurations
  model/                # model IDs, revisions, dtype, positions
data/
  raw/                  # ignored; immutable source downloads
  interim/              # ignored; candidate facts and scores
  processed/            # ignored except tiny fixtures
docs/
  experiment-plan.md
src/panl/
  data/                  # block construction, validation, splits
  models/                # TransformerLens adapters and position resolver
  measures/              # logits, likelihood, commitment
  activations/           # collection and Zarr storage
  analysis/              # factorial models, bootstrap, probes
  interventions/         # projection patching and path controls
  artifacts/             # manifests, hashes, result schemas
tests/
  fixtures/              # tiny synthetic blocks and tokenized prompts
  unit/
  integration/
outputs/                 # ignored; manifests, tables, figures
```

TransformerLens 3 is the default intervention layer because it currently supports both target
model families. Keep a thin project-owned adapter around hook names and semantic positions so
the analysis is not coupled directly to library-internal names.

Target CLI, to be implemented incrementally:

```bash
uv run panl data build --config configs/data/tier1.yaml
uv run panl data validate data/processed/quadruples.parquet
uv run panl collect --config configs/experiment/e0_qwen7b.yaml
uv run panl analyze e1 --run-id RUN_ID
uv run panl intervene e3 --run-id RUN_ID
uv run panl report --run-id RUN_ID
```

## 7. Reproducibility and tests

Before a GPU run, CI must verify:

- every block has exactly four unique `(question_id, answer_id)` cells;
- question and answer identities are balanced across matched/crossed labels;
- no block, question, answer, or held-out family leaks across its declared split;
- prompt rendering and semantic token positions match frozen snapshots per tokenizer;
- the factorial residual is zero for synthetic additive hidden states;
- the residual recovers a planted interaction in synthetic data;
- intervention controls preserve tensor shape, dtype, device, and untouched positions;
- bootstrap and permutation procedures resample at block level;
- a run can be recreated from manifest plus `uv.lock` and git commit.

Use deterministic seeds where supported, but record that GPU kernels may still be
nondeterministic. Publish aggregate results and dataset-building scripts; redistribute raw
datasets/model weights only when their licenses permit it.

## 9. Claim-selection table

| Observed evidence | Allowed conclusion |
|---|---|
| E1 fails after LP controls | Confidence interaction is largely explained by fluency |
| E1 passes, E2 fails held-out | Behavioral interaction exists; no stable PANL subspace identified |
| E2 passes, E3 fails controls | PANL encodes a relational component; causal use not established |
| E1-E3 pass, commitment removal fails | Effect is better characterized as commit-readiness |
| E1-E3 pass all gates in two families | Selective PANL interaction causally contributes to confidence |

This table should appear in the preregistration/runbook and govern the paper title and abstract.
